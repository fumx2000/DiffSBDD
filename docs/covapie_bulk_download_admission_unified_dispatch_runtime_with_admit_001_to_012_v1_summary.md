# CovaPIE unified dispatch runtime with ADMIT_001 to ADMIT_012 v1

This stage implements the formal ADMIT_012 five-envelope runtime adapter and registers it in a new immutable Exact12 single-rule registry. The frozen base is `fc0f89c16b2afd2cac2e5629aa60cf8f962cbdad` (`add CovaPIE ADMIT_012 unified adapter contract design v1`).

The successor identity-re-exports `UnifiedAdmissionRuleEvaluation`, `UnifiedAdmissionDispatchError`, `RESULT_SCHEMA_VERSION`, `RESULT_FIELDS`, `DISPATCH_ERROR_FIELDS`, `DISPATCH_ERROR_CODES`, and `OUTCOME_VOCABULARY` from Exact11. Every ADMIT_001 through ADMIT_011 registry value is the identical predecessor handler object. ADMIT_012 alone is bound to `_evaluate_registered_admit_012`; ADMIT_013 through ADMIT_015 remain known but unregistered.

The ADMIT_012 handler enforces this order: batch must be `None`; evaluation context must be a Mapping; the four policy keys are looked up once in order; download-result context must be a Mapping; stage authorization must be `None`; candidate must be a Mapping; the Exact4 download keys are looked up once until the first missing key; the standalone evaluator is called once; its exact formal result is prevalidated; the committed independent oracle is called once with the same kwargs; all ten values and their exact types must match; only then is the result projected to Exact13.

Candidate records are envelope-only for ADMIT_012. They are never enumerated, copied, or queried. A non-Mapping candidate returns the fixed invalid Exact13 result with empty normalized/validated/consumed-candidate fields and the four policy context items. Missing download-result fields are represented by omitted keywords; the standalone private sentinel is neither imported nor passed.

The source result and oracle result both require exact built-in storage, Exact10 order, successful reconstruction, `ADMIT_012` identity, and `evaluator_io_used is False`. Source/oracle comparison covers every Exact10 field by both exact type and value. Projection accepts only an empty or ordered valid Exact4 pair prefix. Status and SHA remain exact strings; HTTP status and content length become canonical decimal `str(value)` values. `bool`, subclasses, wrong containers, wrong pair shapes, wrong names/order, skipped prefixes, and extra pairs fail closed. No integer pair value enters Exact13.

`validated_candidate_fields` and `consumed_candidate_fields` remain the historical unified Exact13 field names. For ADMIT_012 they carry download-result semantics and do not imply that values came from `candidate_record`.
