# CovaPIE ADMIT_010 leakage-group assignment provenance contract design gate v1

This design-only gate freezes `leakage_group_id` as an exact built-in ASCII string matching `^COVAPIE_LEAKAGE_GROUP_[0-9]{6}$`. The grammar is an explicit V1 choice informed by historical format, capacity, and stability; the five historical values remain an evidence corpus rather than a universal grammar derivation. No trimming, case conversion, repair, normalization, or subclass acceptance is allowed.

The canonical candidate field `leakage_group_id` maps to the historical `final_leakage_group_id` by provider field rename only. Their values must be exactly equal. Assignment IDs, group order, split assignments, single-axis groups, and duplicate identity keys cannot generate or replace the leakage-group ID. Real-provider mapping remains unvalidated.

`LeakageGroupAssignmentProvenanceContractV1` is a frozen, non-slotted, design-only Exact19 value object. It carries exact policy/version/stage values, four syntax-only caller SHA256 attestations, opaque assignment and sample identifiers, ordered unique membership with an exact count, and evidence that assignment passed before any split was written. The pure in-memory design oracle applies the frozen 21-stage precedence and returns only the nine design-classification fields; it does not implement `evaluate_admit_010` or `Admit010EvaluationResult` and performs no I/O.

The deterministic outputs contain 32 provenance-contract rows, 26 mapping/grammar rows, 71 natural truth-matrix rows, the ordered Exact21 committed source boundary, and an 11-row successor issue inventory. Only `LEAKAGE_GROUP_ASSIGNMENT_PROVENANCE_UNRESOLVED` transitions from open to resolved with `leakage_group_assignment_provenance_contract_frozen_v1`; unified coverage remains open from ADMIT_010 through ADMIT_015.

The gate establishes readiness only for the next standalone evaluator-interface implementation step. Provider mapping, formal evaluator/result, adapter, registration, Exact10 runtime, ADMIT_011, aggregate verdicts, real candidate evaluation, bulk download, and training remain unimplemented or prohibited. A feature-semantics audit is still mandatory before formal training; Step12D remains a smoke legality check rather than a final training-feature contract.

Revised1 strengthens evidence checking without changing the design outputs. Production now verifies that every resolved Exact21 source path remains inside the resolved repository root; a parent-directory symlink escape fails closed before any source-byte read. The checker independently performs the same resolved-containment proof and independently freezes the complete Exact104 manifest, including every nested value and ordered list. The provenance contract, oracle, issue transition, readiness state, and all six output files remain byte-identical.

Revised2 makes `contract_version` validation strictly exact-type-first. Any non-exact built-in string, including a hostile `str` subclass or arbitrary object, is rejected before its equality or inequality methods can run, so hostile comparison code cannot execute or leak an exception. This ordering fix does not change the provenance contract, oracle output semantics, truth matrix, Exact104 manifest, issue transition, readiness state, or any of the six design outputs.
